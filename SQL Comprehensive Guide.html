Aggregate Functions

What are Aggregate Functions? Aggregate functions perform calculations on a set of values and return a single value. Common functions include MAX, MIN, SUM, AVG, and COUNT.

Basic MAX() Query

Find the highest salary in the employee table.

Syntax:
SELECT aggregate_function(column_name)
FROM table_name
[WHERE condition];

Query:

SELECT MAX(sal) FROM scott.EMP;
Output:
MAX(SAL)
5000

Explanation: Returns the highest salary (KING's salary of 5000)

GROUP BY with Aggregates

Calculate maximum salary for each department.

Query:

SELECT DEPTNO, MAX(sal) as max_sal FROM scott.EMP GROUP BY DEPTNO;
Output:
DEPTNO MAX_SAL
105000
203000
302850
üíº Real-time Scenario:

A company wants to determine budget allocations by identifying the highest-paid employee in each department for salary benchmarking and compensation planning.

Window Functions

What are Window Functions? Window functions perform calculations across a set of table rows related to the current row, unlike aggregate functions that collapse rows. They maintain row-level detail while performing calculations.

‚úì No Row Collapse

Unlike GROUP BY, window functions don't reduce rows

‚úì Row-Level Detail

Maintain individual row data while calculating

‚úì OVER() Clause

Define the window using OVER clause

‚úì Partition Data

Divide data into partitions for calculation

Window Function vs Aggregate Function

Aggregate (Returns Single Value):

SELECT MAX(sal) FROM scott.EMP; -- Returns: 5000 (single value)

Window Function (Returns Value Per Row):

SELECT e.*, MAX(sal) OVER() as max_salary FROM scott.EMP e;
Output (Sample Rows):
EMPNO ENAME JOB SAL MAX_SALARY
7369SMITHCLERK8005000
7499ALLENSALESMAN16005000
7521WARDSALESMAN12505000
7566JONESMANAGER29755000

Explanation: Every row retains its data while showing the max salary (5000) across all rows.

PARTITION BY Clause

Calculate maximum salary separately for each department.

Query:

SELECT e.*, MAX(sal) OVER(PARTITION BY DEPTNO) as max_salary FROM scott.EMP e;
Output (Sample Rows):
EMPNO ENAME DEPTNO SAL MAX_SALARY
7782CLARK1024505000
7839KING1050005000
7369SMITH208003000
7566JONES2029753000
7499ALLEN3016002850

Explanation: Dept 10 max: 5000, Dept 20 max: 3000, Dept 30 max: 2850

üíº Real-time Scenario:

HR department wants to compare each employee's salary against the highest earner in their department to identify salary gaps and ensure competitive compensation within teams.

Ranking Functions

ROW_NUMBER()

Definition: Assigns a unique sequential integer to rows within a partition, starting from 1. Always unique, no duplicates even for ties.

Key Characteristics:

  • Always assigns unique numbers
  • No duplicate values even for ties
  • Sequential without gaps

Query Example:

SELECT e.*, ROW_NUMBER() OVER(PARTITION BY deptno ORDER BY deptno) as rn FROM scott.emp e;
Output:
EMPNO ENAME DEPTNO SAL RN
7782CLARK1024501
7839KING1050002
7934MILLER1013003
7369SMITH208001
7566JONES2029752
üíº Real-time Scenarios:
  • E-commerce: Get top 3 products by sales in each category
  • Banking: Identify first 5 transactions per customer
  • Education: Select top 2 students per class
  • Healthcare: Find most recent 3 appointments per patient

RANK()

Definition: Assigns a rank to each row within a partition, with gaps in ranking when there are ties. Shares rank for tied values, skips next ranks.

Key Characteristics:

  • Assigns same rank to tied values
  • Skips ranks after ties (leaves gaps)
  • Next rank = previous rank + number of tied rows

Query Example:

SELECT * FROM ( SELECT e.*, RANK() OVER(PARTITION BY DEPTNO ORDER BY sal DESC) as rnk FROM scott.emp e ) x WHERE x.rnk < 4;
Output (Showing ties in Dept 20):
EMPNO ENAME DEPTNO SAL RNK
7788SCOTT2030001
7902FORD2030001
7566JONES2029753

Note: SCOTT and FORD share rank 1 (both have 3000), so next rank is 3 (not 2)

DENSE_RANK()

Definition: Assigns a rank to each row within a partition without gaps, even when there are ties. No gaps in sequence.

Key Characteristics:

  • Assigns same rank to tied values
  • No gaps in ranking sequence
  • Next rank = previous rank + 1 (always consecutive)

Comparison of All Three Ranking Functions:

SELECT e.*, RANK() OVER(PARTITION BY DEPTNO ORDER BY sal DESC) as rnk, DENSE_RANK() OVER(PARTITION BY DEPTNO ORDER BY sal DESC) as dense_rnk, ROW_NUMBER() OVER(PARTITION BY DEPTNO ORDER BY SAL DESC) as rn FROM scott.emp e;
Output (Department 20):
ENAME SAL RNK DENSE_RNK RN
SCOTT3000111
FORD3000112
JONES2975323
ADAMS1100434
SMITH800545
Comparison Analysis:
  • ROW_NUMBER: Always unique (1,2,3,4,5) - useful for pagination
  • RANK: Shares rank, skips next (1,1,3,4,5) - traditional ranking
  • DENSE_RANK: Shares rank, no gaps (1,1,2,3,4) - continuous sequence

Navigation Functions (LAG & LEAD)

LAG() Function

Definition: LAG() accesses data from a previous row in the same result set without using a self-join.
Syntax:
LAG(column_name, offset, default_value) OVER (
  [PARTITION BY partition_expression]
  ORDER BY sort_expression
)

SELECT e.*, LAG(SAL) OVER(PARTITION BY DEPTNO ORDER BY EMPNO) as prev_emp_sal FROM scott.emp e;

Definition: LEAD() accesses data from a subsequent row in the same result set.

Example: Combining LAG and LEAD

Compare each employee's salary with previous employee.

Value Functions

FIRST_VALUE()

Definition: Returns the first value in an ordered set of values within a window frame.

Example: Most Expensive Product per Category

SELECT p.*, FIRST_VALUE(prod_name) OVER( PARTITION BY prod_category ORDER BY PROD_LIST_PRICE DESC ) as most_exp_product FROM sh.products p;
üíº Real-time Scenarios:
  • Retail: Display premium product in each category for upselling
  • Real Estate: Show highest-priced property in each neighborhood
  • Automotive: Display top-of-line model in each vehicle class

LAST_VALUE() with Frame Clause

Important Note: LAST_VALUE() requires proper frame clause to work correctly. Without it, uses default frame which may not give expected results.

Correct Usage with Frame Clause:

SELECT P.*, FIRST_VALUE(PROD_NAME) OVER( PARTITION BY PROD_CATEGORY ORDER BY PROD_LIST_PRICE DESC ) AS MOST_EXP_PRODUCT, LAST_VALUE(PROD_NAME) OVER( PARTITION BY PROD_CATEGORY ORDER BY PROD_LIST_PRICE DESC RANGE BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING ) AS LEAST_EXP_PRODUCT FROM SH.PRODUCTS P;
Frame Boundaries:
  • UNBOUNDED PRECEDING: Start of partition
  • UNBOUNDED FOLLOWING: End of partition
  • CURRENT ROW: Current row
  • n PRECEDING: n rows before current
  • n FOLLOWING: n rows after current

NTH_VALUE() Function

Definition: Returns the nth value in an ordered set of values.

WINDOW Clause for Reusability:

SELECT p.*, FIRST_VALUE(prod_name) OVER w as most_exp_product, LAST_VALUE(prod_name) OVER w as LEAST_EXP_PRODUCT, NTH_VALUE(prod_name, 2) OVER w as second_most_exp_product FROM sh.products p WINDOW w AS ( PARTITION BY PROD_CATEGORY ORDER BY PROD_LIST_PRICE DESC RANGE BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING );
WINDOW Clause Advantages:
  • Reduces code duplication
  • Easier maintenance
  • Improved readability
  • Single definition for multiple functions

NTILE() Function

Definition: Distributes rows in an ordered partition into a specified number of approximately equal groups (buckets/tiles).

Example: Divide Products into 5 Price Buckets

SELECT *, NTILE(5) OVER( PARTITION BY PROD_CATEGORY ORDER BY PROD_LIST_PRICE DESC ) as buckets FROM sh.PRODUCTS;
üíº Real-time Scenarios:
  • Market Segmentation: Divide customers into quartiles (high, medium-high, medium-low, low value)
  • Performance Rating: Categorize employees into performance tiers
  • Risk Assessment: Group investments into risk buckets
  • Inventory Management: Categorize products by turnover rate

SQL Joins

What are Joins? SQL joins combine rows from two or more tables based on related columns between them.

INNER JOIN

Returns matching rows from both tables

LEFT JOIN

All rows from left, matching from right

RIGHT JOIN

All rows from right, matching from left

FULL JOIN

All rows when match in either table

CROSS JOIN

SELF JOIN

Table joined with itself

Key Constraints

What are Key Constraints? Key constraints enforce data integrity rules at the database level, ensuring data accuracy and consistency.

PRIMARY KEY

Uniquely identifies each record. Cannot be NULL. Only one per table.

FOREIGN KEY

Links tables and enforces referential integrity with parent table.

UNIQUE

Ensures all values in column are distinct. Allows NULL values.

NOT NULL

Prevents NULL values from being inserted into column.

CHECK

Validates data against specific condition or business rule.

PRIMARY KEY

Creating Primary Key:

CREATE TABLE employees ( emp_id NUMBER PRIMARY KEY, emp_name VARCHAR2(50) NOT NULL, email VARCHAR2(100) );

Composite Primary Key:

CREATE TABLE order_details ( order_id NUMBER, product_id NUMBER, quantity NUMBER, CONSTRAINT pk_order_details PRIMARY KEY (order_id, product_id) );

FOREIGN KEY

Creating Foreign Key:

CREATE TABLE employees ( emp_id NUMBER PRIMARY KEY, emp_name VARCHAR2(50), dept_id NUMBER, CONSTRAINT fk_dept FOREIGN KEY (dept_id) REFERENCES departments(dept_id) );
Referential Integrity Actions:
  • ON DELETE CASCADE: Delete child rows when parent deleted
  • ON DELETE SET NULL: Set foreign key to NULL when parent deleted
  • ON UPDATE CASCADE: Update child rows when parent updated
  • ON DELETE RESTRICT: Prevent deletion if child rows exist (default)

CHECK Constraint

Validates data based on specified business rules.

CREATE TABLE employees ( emp_id NUMBER PRIMARY KEY, emp_name VARCHAR2(50) NOT NULL, age NUMBER CHECK (age >= 18 AND age <= 65), salary NUMBER CHECK (salary > 0), email VARCHAR2(100) CHECK (email LIKE '%@%') );
üíº Real-time Examples:
  • Retail: CHECK (quantity > 0) for order items
  • Banking: CHECK (balance >= 0) for savings accounts
  • Education: CHECK (grade BETWEEN 0 AND 100)
  • HR: CHECK (hire_date <= SYSDATE) prevent future dates

Indexing

What is an Index? An index is a database object that improves the speed of data retrieval operations at the cost of additional storage and slower write operations. Think of it like a book index - jump directly to relevant pages instead of reading the entire book.

Types of Indexes

B-Tree Index

Default, balanced tree structure (most common)

Bitmap Index

For low-cardinality columns (few distinct values)

Unique Index

Enforces uniqueness constraint

Composite Index

Multiple columns indexed together

Creating Indexes

Basic Index:

CREATE INDEX idx_emp_name ON scott.emp(ename);

Unique Index:

CREATE UNIQUE INDEX idx_emp_email ON employees(email);

Composite Index:

CREATE INDEX idx_emp_dept_sal ON scott.emp(deptno, sal);

Function-Based Index:

CREATE INDEX idx_upper_name ON scott.emp(UPPER(ename));

When to Use Indexes

‚úì Good Candidates for Indexing:
  • Columns frequently used in WHERE clauses
  • Columns used in JOIN conditions
  • Columns used in ORDER BY or GROUP BY
  • Foreign key columns
  • Columns with high selectivity (many unique values)
‚úó Avoid Indexing:
  • Small tables (full scan is faster)
  • Columns with low selectivity (gender, status)
  • Columns frequently updated (index maintenance overhead)
  • Tables with heavy INSERT/UPDATE operations

Index Performance Impact

Operation Without Index With Index
SELECT (1 row from 1M rows) 0.8 seconds 0.001 seconds
SELECT (100 rows) 0.9 seconds 0.01 seconds
INSERT 0.01 seconds 0.02 seconds
UPDATE 0.02 seconds 0.04 seconds

Conclusion: Indexes dramatically improve SELECT performance but add overhead to INSERT/UPDATE operations.

Composite Index Column Order

Rule: Place most selective column first in composite index.

Query:

SELECT * FROM employees WHERE dept_id = 10 AND salary > 5000;

Good Composite Index (dept_id first):

CREATE INDEX idx_dept_sal ON employees(dept_id, salary); -- dept_id first (more selective), then salary
üíº Real-time Scenarios:
  • E-commerce: Index on (category_id, price) for product searches
  • Banking: Index on (account_id, trans_date) for transaction history
  • Healthcare: Index on (doctor_id, appt_date) for appointment scheduling

Normalization

What is Normalization? Normalization is the process of organizing data in a database to reduce redundancy and improve data integrity by decomposing tables into smaller tables and defining relationships.

Normalization Goals

  • Eliminate data redundancy
  • Ensure data dependencies make sense
  • Prevent insertion, update, and deletion anomalies
  • Simplify data maintenance

Normal Forms Summary

Normal Form Rule Problem Solved
1NF Atomic values only Multi-valued attributes
2NF No partial dependency Partial dependencies on composite keys
3NF No transitive dependency Transitive dependencies
BCNF Stricter 3NF Remaining anomalies

First Normal Form (1NF)

Rules:
  • Each column must contain atomic (indivisible) values
  • Each column must contain values of single type
  • Each record must be unique (have primary key)
  • No repeating groups or arrays

Denormalization

Definition: Intentionally introducing redundancy to improve query performance.

When to Denormalize:

  • Read-heavy workloads (data warehouses)
  • Complex joins impact performance
  • Aggregated data frequently accessed

Query Optimization

What is Query Optimization? The process of improving SQL query performance by reducing execution time and resource consumption.

Key Performance Metrics

‚è±Ô∏è Execution Time

How long query takes to complete

üíæ I/O Operations

Disk reads and writes

üî• CPU Usage

Processing power consumed

üß† Memory Usage

RAM allocation during execution

Top 10 Optimization Techniques

1. Use Indexes Effectively

‚ùå Slow (Function prevents index use):

SELECT * FROM employees WHERE UPPER(email) = 'JOHN@EXAMPLE.COM';

‚úì Fast (Uses function-based index):

CREATE INDEX idx_email_upper ON employees(UPPER(email)); SELECT * FROM employees WHERE UPPER(email) = 'JOHN@EXAMPLE.COM';

2. Avoid SELECT *

‚ùå Slow (Retrieves all columns):

SELECT * FROM employees WHERE dept_id = 10;

‚úì Fast (Only needed columns):

SELECT emp_id, emp_name, salary FROM employees WHERE dept_id = 10;

3. Use WHERE Instead of HAVING

‚ùå Slow (Filters after aggregation):

SELECT dept_id, COUNT(*) FROM employees GROUP BY dept_id HAVING dept_id = 10;

‚úì Fast (Filters before aggregation):

SELECT dept_id, COUNT(*) FROM employees WHERE dept_id = 10 GROUP BY dept_id;

4. Use EXISTS Instead of IN

‚ùå Slow (Subquery executes fully):

SELECT * FROM employees WHERE dept_id IN ( SELECT dept_id FROM departments WHERE location = 'New York' );

‚úì Fast (Stops at first match):

SELECT * FROM employees e WHERE EXISTS ( SELECT 1 FROM departments d WHERE d.dept_id = e.dept_id AND d.location = 'New York' );

5. Avoid Functions in WHERE Clause

‚ùå Slow (Function prevents index):

SELECT * FROM employees WHERE YEAR(hire_date) = 2023;

‚úì Fast (Uses index on hire_date):

SELECT * FROM employees WHERE hire_date >= '2023-01-01' AND hire_date < '2024-01-01';

Query Optimization Checklist

  • Proper indexes on frequently queried columns
  • Avoid SELECT * (specify columns)
  • Use WHERE instead of HAVING for filtering
  • Avoid functions on indexed columns in WHERE
  • Use EXISTS instead of IN for subqueries
  • Use UNION ALL instead of UNION when possible
  • Limit result sets with FETCH/TOP/LIMIT
  • Use appropriate data types
  • Keep statistics updated
  • Monitor and analyze execution plans

Real-time Optimization Example: E-commerce Product Search

Before Optimization:

SELECT * FROM products WHERE LOWER(product_name) LIKE '%laptop%'; -- Full table scan, function on indexed column

Performance: 2.5 seconds for 1M products

After Optimization:

CREATE INDEX idx_product_name_lower ON products(LOWER(product_name)); SELECT product_id, product_name, price FROM products WHERE LOWER(product_name) LIKE 'laptop%'; -- Uses index, specific columns, optimized LIKE pattern

Performance: 0.02 seconds (125x faster!)

Views

What is a View? A view is a virtual table based on the result set of a SQL query. It contains rows and columns from one or more tables but doesn't store data itself.

Virtual Table

No physical storage, defined by query

Simplifies Queries

Hides complex joins and logic

Security Layer

Restricts access to sensitive data

Current Data

Always shows latest data from base tables

Creating Views

Simple View (Employee + Department):

CREATE VIEW emp_dept_view AS SELECT e.empno, e.ename, e.sal, d.dname, d.loc FROM scott.emp e INNER JOIN scott.dept d ON e.deptno = d.deptno;

Query the View:

SELECT * FROM emp_dept_view WHERE dname = 'SALES';

Complex View with Aggregations

Department Salary Summary View:

View Security Features

Restrict Column Access (Hide Sensitive Data):

CREATE VIEW emp_public_view AS SELECT empno, ename, job, deptno FROM scott.emp; -- Excludes salary, commission (sensitive data)

Restrict Row Access (Department-Specific View):

CREATE VIEW sales_dept_view AS SELECT * FROM scott.emp WHERE deptno = 30; -- Only SALES department visible

WITH CHECK OPTION (Enforce View Constraints):

CREATE VIEW high_salary_view AS SELECT * FROM scott.emp WHERE sal > 2000 WITH CHECK OPTION; -- This will fail (salary too low) INSERT INTO high_salary_view VALUES (9999, 'TEST', 'CLERK', 7839, SYSDATE, 1500, NULL, 10);

Materialized Views

Difference from Regular Views:
  • Regular View: Virtual, no storage, runs query each time
  • Materialized View: Stores data physically, refreshed periodically

Create Materialized View:

CREATE MATERIALIZED VIEW emp_dept_mv BUILD IMMEDIATE REFRESH COMPLETE ON DEMAND AS SELECT e.empno, e.ename, e.sal, d.dname, d.loc FROM scott.emp e INNER JOIN scott.dept d ON e.deptno = d.deptno;
Advantages:
  • Faster queries (data pre-computed and stored)
  • Reduces load on source tables
  • Excellent for data warehousing and reporting
Disadvantages:
  • Requires storage space
  • Data may be stale (until refresh)
  • Maintenance overhead

CREATE VIEW customer_order_summary AS SELECT c.customer_id, c.customer_name, COUNT(o.order_id) as total_orders, SUM(o.order_amount) as total_spent, MAX(o.order_date) as last_order_date FROM customers c LEFT JOIN orders o ON c.customer_id = o.customer_id GROUP BY c.customer_id, c.customer_name;
üíº Banking: Account Balance View
CREATE VIEW account_balance_view AS SELECT a.account_number, a.account_type, a.opening_balance + COALESCE(SUM(t.amount), 0) as current_balance FROM accounts a LEFT JOIN transactions t ON a.account_id = t.account_id GROUP BY a.account_number, a.account_type, a.opening_balance;
